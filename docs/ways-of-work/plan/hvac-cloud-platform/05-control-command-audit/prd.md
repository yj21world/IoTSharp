# Feature PRD：控制命令与审计模块

## 1. Feature 名称

控制命令与审计模块

## 2. 所属 Epic

- 父级产品需求：[暖通空调云端物联网平台长期产品需求](../epic.md)
- 父级架构文档：[暖通空调云端物联网平台架构规格](../arch.md)
- 前置模块：[网关与设备接入模块](../01-gateway-device-access/prd.md)

## 3. 目标

### 问题

暖通空调系统运维需要向现场设备下发控制命令，如启动/停止水泵、设置频率、开关阀门、复位故障等。当前系统已有 MQTT RPC 通道（`RpcClient` / `RpcController`），支持基于 MQTT 的请求/响应模式，但缺少命令生命周期追踪、操作权限校验、审计日志记录和结果反馈。一次"下发命令"如果只等同于一次 MQTT Publish，无法满足生产环境对安全性、可追溯性和可诊断性的要求。

### 解决方案

第一阶段在现有 RPC 通道基础上做最简版增强：补齐操作日志记录和基本权限校验。允许授权用户通过 HTTP API 或 MQTT 向设备或网关下发控制命令，追踪发送、响应和超时状态。完整状态机（Pending → Sent → Executed → Confirmed / Failed / Timeout）推迟到后续阶段。

### 影响

控制命令是暖通平台的核心业务闭环之一。完成后，平台不仅能"看"数据，还能"控"设备，形成完整的采集与控制双向通道。

## 4. 使用场景

- 授权用户在平台上选择目标设备和控制动作（如"启动 1# 水泵"），平台通过 MQTT 下发命令。
- 对于透传网关场景，云端通过网关向子设备下发 Modbus 写线圈或写寄存器命令。
- 对于边缘网关场景，云端下发结构化 JSON 控制命令。
- 运维人员查看控制命令历史记录，按设备、时间、操作人和状态过滤。
- 故障排查时查看每条命令的详细信息和执行结果。

## 5. 用户故事

- 作为授权用户，我希望能通过平台向指定设备下发控制命令（如启动、停止、设置频率），以便远程操控现场设备。
- 作为授权用户，我希望能在下发命令前确认系统已校验我的操作权限，以避免误操作或越权操作。
- 作为运维人员，我希望能查看控制命令的历史记录，包括操作人、时间、目标设备、命令内容和执行结果。
- 作为系统管理员，我希望能追踪每条命令的状态（已发送、成功、失败、超时），以便判断控制链路是否正常。
- 作为调试人员，我希望透传网关能向 Modbus 设备下发写命令，以便在现场测试和调试阶段验证设备响应。

## 6. 需求

### 6.1 功能需求

- 系统必须支持通过 HTTP API 向设备下发控制命令。
- 系统必须支持通过 MQTT RPC 通道向设备下发控制命令。
- 控制命令必须包含命令类型（启动、停止、复位、设定值、频率设置等）。
- 控制命令必须包含目标方法名和参数。
- 系统必须在命令下发前校验操作人权限（至少校验租户/客户归属）。
- 系统必须记录每次命令操作的完整信息（操作人、时间、目标设备、命令类型、参数、状态、结果、耗时）。
- 系统必须支持接收设备 RPC 响应并更新命令状态。
- 系统必须支持命令超时检测（默认 5000ms）。
- 系统必须支持查询控制命令历史记录，支持按设备、操作人、命令类型、状态和时间范围过滤。
- 对于透传网关场景，系统必须支持通过网关向子设备下发 Modbus 写命令。
- 对于边缘网关场景，系统必须支持下发结构化 JSON 控制命令。
- 新增/调整 API 必须遵循 `ApiResult<T>`，列表/查询使用 `{ total, rows }`。

### 6.2 当前已具备能力

- `RpcClient` 已支持 MQTT RPC 请求/响应模式（含超时机制）。
- `RpcController`（MQTT）已支持 RPC 消息处理和流规则联动。
- `AuditLog` 实体已存在，可用于命令审计。
- MQTT RPC Topic 约定已定义：`devices/{devname}/rpc/request/{requestId}` 和 `devices/{devname}/rpc/response/{requestId}`。

### 6.3 后续补齐需求

- 创建 `CommandLog` 实体，记录命令的完整生命周期信息（操作人、设备、方法、参数、状态、结果、耗时）。
- 创建 `CommandService` 和 `CommandController`，提供命令下发和查询 API。
- 实现基础权限校验（操作人与目标设备同租户/客户）。
- 补齐 HTTP API 的命令下发端点。
- 定义暖通常用命令类型枚举（启动、停止、复位、设定值、写线圈、写寄存器等）。
- 增加控制命令的最小测试用例，覆盖权限校验、命令下发、响应处理和日志查询。

### 6.4 非功能需求

- 控制命令下发必须记录审计信息，不得静默执行。
- 控制命令权限校验失败必须返回明确错误，不得绕过。
- 命令日志必须记录操作人真实身份（来自 JWT Claims）。
- 第一阶段暂不实现完整命令审批流程和命令模板机制。
- 透传网关写命令场景需要注意 Modbus 写操作的安全性，后续阶段需增加白名单校验。

## 7. 验收标准

### 7.1 命令下发

- Given 授权用户已登录
  When 调用命令下发 API 指定设备、命令类型和参数
  Then 系统应校验权限、记录日志、通过 MQTT RPC 下发命令，并返回命令 ID 和状态。

- Given 用户无目标设备的操作权限
  When 调用命令下发 API
  Then 系统应返回权限拒绝错误，不得下发命令。

### 7.2 命令响应

- Given 命令已下发且设备在超时时间内返回 RPC 响应
  When 系统收到响应
  Then 系统应更新命令状态为成功、记录结果数据和耗时。

- Given 命令已下发但超时未收到响应
  When 超时触发
  Then 系统应更新命令状态为超时。

### 7.3 命令查询

- Given 运维人员查询命令历史
  When 按设备、时间范围和状态过滤
  Then 系统应返回分页命令日志列表，包含操作人、命令内容、状态和耗时。

- Given 运维人员查看单条命令详情
  When 查询指定命令 ID
  Then 系统应返回命令的完整信息，包括请求参数和响应结果。

## 8. 不在范围内

- 不在本模块实现完整控制命令状态机（Pending → Sent → Executed → Confirmed）。第一阶段只区分 Sent / Success / Failed / Timeout。
- 不在本模块实现命令审批流程。
- 不在本模块实现控制命令模板机制。
- 不在本模块实现 Modbus 写命令的白名单安全校验。
- 不在本模块实现批量命令下发。
- 不在本模块实现命令重试和自动恢复。

## 9. 依赖与后续关系

- 依赖"网关与设备接入模块"提供设备认证、归属和 MQTT 通道。
- 依赖"透传网关 Modbus 采集模块"的 MQTT 传输通道用于透传网关写命令。
- 依赖 ASP.NET Core Identity + JWT 提供用户认证。
- 本模块是"告警与规则链暖通化模块"的控制动作执行基础。

## 10. 当前状态判断

**状态：RPC 通道已有，需补齐命令记录、权限校验和查询 API。**

当前 MQTT RPC 通道已工作，但缺少结构化的命令日志、权限校验和 HTTP 命令 API。第一阶段应以最小改动补齐这三个缺口，不做完整状态机和命令模板。
