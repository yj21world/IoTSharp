# Feature PRD：实时数据与历史数据查询模块

## 1. Feature 名称

实时数据与历史数据查询模块

## 2. 所属 Epic

- 父级产品需求：[暖通空调云端物联网平台长期产品需求](../epic.md)
- 父级架构文档：[暖通空调云端物联网平台架构规格](../arch.md)
- 数据来源：[网关与设备接入模块](../01-gateway-device-access/prd.md)、[透传网关 Modbus 采集模块](../02-transparent-modbus-collection/prd.md)、[边缘网关 JSON 接入模块](../03-edge-json-ingestion/prd.md)

## 3. 目标

### 问题

暖通空调运维人员需要实时查看设备运行状态（温度、压力、频率、功率等最新值），并查询历史趋势曲线来判断设备运行是否正常。当前系统已具备 `TelemetryLatest` 最新值表、`TelemetryData` TimescaleDB 超表和 `IStorage` 查询接口，但面向用户的查询 API 完整性、参数校验、权限过滤和响应格式有待统一和完善。此外，数据的写入链路（采集模块 → CAP → IStorage）已经工作，查询模块不需要关心数据如何入库。

### 解决方案

基于现有 `IStorage` 接口和 TimescaleDB 存储实现，提供统一的设备遥测和属性查询 API。支持按设备、遥测键、时间范围和聚合函数查询最新值和历史趋势数据，遵循统一 `ApiResult<T>` 返回格式并校验设备归属。

### 影响

运维人员和调试工程师可以在平台上直接查看设备实时数据和历史曲线，无需依赖第三方工具。统一的查询 API 也为后续能耗统计、告警规则和前端仪表盘提供标准数据入口。

## 4. 使用场景

- 运维人员打开设备详情页，查看该设备所有遥测点位的最新值和更新时间。
- 运维人员选择一个遥测键（如"1#水泵 频率"）和最近 24 小时，查看历史趋势曲线。
- 运维人员同时查看一台设备的多个遥测键（温度、湿度、压力），多条曲线叠加展示。
- 运维人员查询过去 7 天的平均功率，按小时聚合。
- 调试人员选择指定遥测键的最新值，确认采集链路是否正常工作。

## 5. 用户故事

- 作为运维人员，我希望能查看设备所有遥测点位的最新值，以便快速了解设备当前运行状态。
- 作为运维人员，我希望能查看指定遥测键在过去一段时间内的历史数据，以便分析趋势和排查异常。
- 作为运维人员，我希望能对历史数据进行聚合（平均值、最大值、最小值），以便查看日/周/月级别的统计。
- 作为调试人员，我希望能按遥测键名过滤最新值，以便只关注我关心的点位。
- 作为平台使用方，我希望查询结果中包含正确的时间戳和数据类型标记，以便前端正确渲染。

## 6. 需求

### 6.1 功能需求

- 系统必须支持查询指定设备的所有遥测点位最新值。
- 系统必须支持查询指定设备的指定遥测键最新值（逗号分隔多键）。
- 系统必须支持查询指定设备的指定遥测键历史数据（时间范围必选）。
- 历史查询必须支持聚合函数：Mean（平均）、Max（最大）、Min（最小）、Sum（求和）、First（首值）、Last（末值）。
- 历史查询必须支持时间窗口参数（every），如 1 小时、1 天。
- 系统必须支持多键并发查询（`Task.WhenAll`）。
- 查询结果必须包含遥测键名、时间戳、值和数据类型。
- 查询时必须校验设备归属（租户/客户级别），无权限用户不得查看其他租户设备数据。
- 系统必须支持查询指定设备的属性最新值。
- 新增/调整 API 必须遵循 `ApiResult<T>`，列表/查询使用 `{ total, rows }`。

### 6.2 当前已具备能力

- `IStorage` 接口已定义：`GetTelemetryLatest(deviceId)`、`GetTelemetryLatest(deviceId, keys)`、`LoadTelemetryAsync(deviceId, keys, begin, end, every, aggregate)`。
- `TimescaleDBStorage` 已实现完整查询逻辑（EF Core 读最新值 + 原生 SQL `time_bucket` 聚合查询）。
- TimescaleDB 超表已自动创建索引：`(KeyName, DateTime DESC)`、`(DataSide, DateTime DESC)`、`(Type, DateTime DESC)`。
- `TelemetryDataDto` 已定义查询返回结构。
- CAP EventBus → `IStorage.StoreTelemetryAsync` 写入链路已工作。

### 6.3 后续补齐需求

- 审查 `DevicesController` 中遥测查询端点完整性，补充缺失的 `GetTelemetryLatest` 和 `GetTelemetryHistory` 端点。
- 统一查询 API 的参数校验和默认值（begin 默认 24 小时前，end 默认当前时间）。
- 统一查询结果的返回格式，确保与 API 规范一致。
- 补充设备归属校验逻辑（`IJustMy` 或等价机制）。
- 考虑最新值查询是否需要缓存层（当前 DB 直接查询已满足第一阶段需求）。
- 增加遥测查询的最小测试用例，覆盖最新值查询、历史查询、聚合查询和权限过滤。

### 6.4 非功能需求

- 单设备最新值查询（全部键）响应时间应 < 500ms（10 网关量级）。
- 历史数据查询响应时间应 < 2s（单键、24 小时范围、无聚合）。
- 聚合查询利用 TimescaleDB `time_bucket` 原生函数，不得在应用层做聚合。
- 查询结果中的时间戳统一使用 UTC。
- 查询不得绕过租户/客户数据隔离。

## 7. 验收标准

### 7.1 最新值查询

- Given 已登录用户有设备访问权限
  When 查询指定设备的最新遥测值（不指定 keys）
  Then 系统应返回该设备所有遥测键的最新值和更新时间。

- Given 已登录用户
  When 查询指定设备的 `temperature,humidity` 最新值
  Then 系统应仅返回这两个键的最新值。

### 7.2 历史数据查询

- Given 已登录用户
  When 查询指定设备的 `temperature` 过去 24 小时历史数据（无聚合）
  Then 系统应返回时间序列数据列表，按时间升序排列。

- Given 已登录用户
  When 查询指定设备的 `power` 过去 7 天数据，按 1 小时聚合取平均值
  Then 系统应返回每小时一个数据点，值为该小时内所有数据的平均值。

### 7.3 权限过滤

- Given 已登录用户不属于目标设备的租户
  When 查询该设备的遥测数据
  Then 系统应返回无权限错误。

## 8. 不在范围内

- 不在本模块实现跨设备聚合查询（如"冷站所有温度传感器的平均值"）。
- 不在本模块实现数据导出（CSV / Excel）。
- 不在本模块实现实时数据推送（WebSocket / SSE）。
- 不在本模块实现数据缓存层。
- 不在本模块处理数据写入逻辑（数据写入由采集模块和 CAP EventBus 处理）。

## 9. 依赖与后续关系

- 依赖遥测存储链路（TimescaleDB + `IStorage` + CAP EventBus）提供数据查询能力。
- 依赖"网关与设备接入模块"提供设备归属和权限信息。
- 本模块是"能耗统计模块"的数据查询基础。
- 本模块是前端数据展示页面的 API 基础。

## 10. 当前状态判断

**状态：存储和查询核心已就绪，需补齐查询 API 和权限校验。**

`IStorage` 接口和 TimescaleDB 实现已经工作，但 Controller 层的查询端点可能不完整。第一阶段应重点审查和补充 HTTP 查询 API，统一参数和返回格式，补上权限过滤，不需要改动存储层。
