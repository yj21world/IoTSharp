# Feature PRD：暖通业务设备建模模块

## 1. Feature 名称

暖通业务设备建模模块

## 2. 所属 Epic

## 3. 目标

### 问题

当前系统中，现场底层接入对象是 Modbus 从机、IO 模块、电表和传感器等通信层实体。现场运维人员关心的却是业务层对象——"1#水泵"、"2#主机"、"冷冻水系统"。一个业务设备可能聚合多个底层数据源：水泵需要同时关注变频器数据、电表数据和出口压力；主机需要聚合多个 Modbus 从机的压缩机、冷凝器和蒸发器数据。如果平台只展现零散的通信点位，运维人员无法快速理解设备整体状态。

### 解决方案

在现有 `Asset`、`AssetRelation`、`Device` 模型基础上，构建暖通业务设备视图。业务设备与底层通信设备分离建模，一个业务设备可关联多个底层数据源，形成"业务设备 → 数据源 → 遥测点位"的聚合关系。支持暖通领域常用的设备分类：主机、水泵、冷却塔、阀门、风柜、电表、传感器和 IO 模块。

### 影响

完成此模块后，运维人员看到的将是符合暖通运维习惯的业务设备视图，而非零散的通信层点位。这为后续能耗统计、告警规则和运维报表提供真正的"设备"语义基础。

## 4. 使用场景

- 运维人员创建"1#水泵"业务设备，关联变频器（频率、电流）和电表（功率、电量）底层数据源。
- 运维人员创建"1#冷站"系统，下挂主机、水泵、冷却塔等子设备。
- 运维人员在业务设备视图下查看所有关联数据源的实时数据聚合。
- 告警规则基于业务设备维度设置（如"1#水泵 频率 > 50Hz 持续 5 分钟"）。

## 5. 用户故事

- 作为运维人员，我希望能创建暖通业务设备（如水泵、主机），并关联底层 Modbus 设备和传感器，以便看到业务层设备视图。
- 作为运维人员，我希望能定义业务设备的层级关系（如冷站 → 系统 → 设备），以便按层级导航和管理。
- 作为调试工程师，我希望一个底层 Modbus 设备的数据可以被多个业务设备复用（如电表数据同时用于水泵和主机能耗计算）。
- 作为平台管理人员，我希望能定义业务设备类型及其默认关联的数据源模板。

## 6. 需求

### 6.1 功能需求（目标态）

- 系统必须支持创建暖通业务设备，并选择业务设备类型（主机、水泵、冷却塔、阀门、风柜、风机盘管、电表、传感器、IO 模块）。
- 系统必须支持业务设备与底层 `Device` / `Gateway` 建立多对多或一对多关联。
- 系统必须支持业务设备的层级关系（冷站 → 系统 → 设备），树形或图结构。
- 系统必须支持从业务设备视角查看聚合后的实时数据。
- 业务设备必须复用现有 `Asset` / `AssetRelation` 或在此基础上扩展新实体。
- 业务设备类型必须支持定义默认数据源模板。

### 6.2 待确认问题

- `Asset` / `AssetRelation` 的当前语义是否可以直接承载暖通业务设备？还是需要新建实体？
- 业务设备的层级是物理层级（冷站→设备）还是逻辑分组（系统→设备组→设备）？
- 一个底层数据源（如电表）是否可以被多个业务设备引用？如果可以，数据的归属如何处理？
- 业务设备的数据聚合是实时关联查询还是通过配置的映射关系预计算？

### 6.3 非功能需求（目标态）

- 业务设备与底层设备的关联不得破坏现有采集和遥测链路。
- 业务设备查询性能不应因关联层级过深而显著下降。

## 8. 不在范围内（第一阶段）

- 不在第一阶段实现完整的业务设备建模。
- 不在第一阶段实现业务设备与底层设备的聚合查询。
- 不在第一阶段实现设备组/系统概念。

## 9. 第一阶段可做的准备

- 在 `Device` 实体中预留 `HVACDeviceType` 字段或等效标记。
- 在 `Produce`（产品）模型中支持暖通设备类型标记。
- 保持 `Asset` / `AssetRelation` 实体不变，不破坏后续扩展路径。

## 10. 当前状态判断

**状态：第二阶段模块，第一阶段暂不实施。建议在阶段一完成通信层闭环后再启动。**

当前 `Asset` / `AssetRelation` 实体已存在，暖通业务设备建模将在其基础之上扩展或新建实体。第一阶段先跑通数据采集与控制闭环，为业务设备建模积累真实设备数据和场景。
