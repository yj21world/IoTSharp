# Feature PRD：能耗统计模块

## 1. Feature 名称

能耗统计模块

## 2. 所属 Epic

- 父级架构文档：[暖通空调云端物联网平台架构规格](../arch.md)

## 3. 目标

### 问题

暖通空调系统是建筑能耗的主要组成部分，运维方需要按站点、系统、设备类型和时间周期统计电耗、水耗、冷量等能源数据。当前平台已经具备遥测数据存储能力，但缺少将原始遥测数据聚合为能耗统计指标的计算、存储和查询能力。能耗统计不是简单的"查历史数据然后求和"，它需要处理累计量（电表读数）、瞬时量（功率）和衍生指标（COP、单位冷量电耗）。

### 解决方案

基于历史遥测数据，建立独立的能耗统计计算和存储体系。按小时、日、月、年等周期预计算或实时聚合能耗指标，提供站点级、系统级、设备级的多维度能耗对比和趋势查询。首期优先支持电耗统计，后续扩展水耗、冷量和能效指标。

### 影响

能耗统计是暖通平台从"监控工具"升级为"运维管理工具"的关键能力。它使运维方能够量化能源消耗、识别能效异常、评估节能措施效果。

## 4. 使用场景

- 运维人员查看某站点过去 30 天的日总电耗趋势。
- 运维人员比较同一站点下不同主机的水泵电耗占比。
- 运维人员查询某台设备的月度累计运行时长。
- 管理人员导出月度能耗报表（站点级、设备级）。
- 后续告警模块基于能耗异常触发预警（如"今日电耗高于昨日同期 20%"）。

## 5. 用户故事

- 作为运维人员，我希望能查看站点过去 30 天的日总电耗趋势，以便评估能耗变化。
- 作为运维人员，我希望能按设备类型（主机、水泵、冷却塔）分类统计电耗，以便识别主要能耗来源。
- 作为管理人员，我希望能导出月度能耗报表，以便向客户或上级汇报。
- 作为调试工程师，我希望能确认能耗数据的来源点位配置正确，以便保证统计准确性。

## 6. 需求

### 6.1 功能需求（目标态）

- 系统必须支持电耗数据分类（按站点、设备类型、设备、能源类型）。
- 系统必须支持按小时、日、月、年周期统计电耗。
- 系统必须支持设备级、系统级、站点级能耗对比查询。
- 系统必须支持能耗趋势图表查询 API。
- 首期优先支持电耗，后续扩展水耗、冷量和运行时长统计。
- 能耗数据来源必须可追溯（关联到具体的遥测键和采集点位）。
- 新增/调整 API 必须遵循 `ApiResult<T>`，列表/查询使用 `{ total, rows }`。

### 6.2 实现策略（待确认）

- 统计计算方式：实时聚合（每次查询时从 TelemetryData 聚合） vs 预计算（Quartz 定时任务提前汇总到独立统计表）。
- 累计量（电表读数）的差值计算：需要处理电表归零、换表和异常跳变。
- TimescaleDB continuous aggregate 是否适用于能耗统计场景（自动维护物化视图）。

### 6.3 非功能需求（目标态）

- 能耗统计不得影响原始遥测数据的写入性能。
- 统计数据的精度和可追溯性优先于查询速度。
- 对电表归零、换表和异常跳变需要明确的处理策略。

## 7. 验收标准

待第三阶段详细设计时定义。

## 8. 不在范围内（第一阶段）

- 不在第一阶段实现任何能耗统计功能。
- 不在第一阶段建设独立的能耗统计表。
- 不在第一阶段实现 COP、EER 等高级能效指标。
- 不在第一阶段实现能耗报表导出。

## 9. 第一阶段可做的准备

- 在遥测键命名约定中为能耗相关点位预留前缀（如 `energy_kwh_*`、`power_kw_*`）。
- 在设备类型模板中支持能耗点位标记。
- TimescaleDB 超表已为后续聚合查询做好准备，不需要额外基础设施改动。

## 10. 当前状态判断

**状态：第三阶段模块，第一阶段暂不实施。建议在至少积累数月真实遥测数据后再启动。**

能耗统计模块的核心设计决策（实时聚合 vs 预计算、累计量处理、TimescaleDB continuous aggregate 的适用性）需要在有数据积累后再做验证。第一阶段重点是保证遥测数据正确、稳定入库。
