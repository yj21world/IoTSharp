# Feature PRD：边缘网关 JSON 接入模块

## 1. Feature 名称

边缘网关 JSON 接入模块

## 2. 所属 Epic

- 父级产品需求：[暖通空调云端物联网平台长期产品需求](../epic.md)
- 父级架构文档：[暖通空调云端物联网平台架构规格](../arch.md)
- 前置模块：[网关与设备接入模块](../01-gateway-device-access/prd.md)

## 3. 目标

### 问题

暖通现场存在具备边缘采集计算能力的网关，这类网关在本地主动采集 Modbus 设备、IO 设备和传感器数据后，将结果组装为 JSON 格式上传到云端。与透传网关不同，边缘网关的采集逻辑在本地执行，云端的职责不是生成 Modbus 请求，而是接收、校验、映射和存储 JSON 数据。当前系统已具备 MQTT ThingsBoard 兼容网关协议和 HTTP 遥测入口，但缺乏统一的边缘 JSON 接入规范、字段映射配置和格式诊断能力。

### 解决方案

在现有 HTTP/MQTT 接入层和 ThingsBoard 兼容协议基础上，定义边缘网关 JSON 上传的标准格式、校验规则、字段映射和错误诊断机制。支持单设备直接上传和网关批量上传两种模式，使云端能够正确接收并存储来自各种边缘网关的数据。

### 影响

完成此模块后，边缘采集网关可以直接通过 HTTP 或 MQTT 上传 JSON 数据到云端，无需为每种网关实现定制解析逻辑。数据通过统一的字段映射和校验管道后进入遥测存储链路，与透传网关采集的数据享受相同的查询、告警和统计能力。

## 4. 使用场景

- 边缘网关在本地采集 Modbus/IO 设备数据，组装为 JSON 后上传。
- 边缘网关同时管理多个子设备，通过批量接口一次性上传所有子设备数据。
- 现场调试人员需要查看 JSON 上传是否成功、失败原因和原始数据上下文。
- 网关厂商不同、JSON 字段命名不同，需要通过字段映射统一到平台遥测键。

## 5. 用户故事

- 作为边缘网关开发者，我希望按照平台定义的 JSON 格式上传数据，以便网关能稳定接入云端。
- 作为现场调试人员，我希望边缘网关上传的 JSON 字段能映射到平台遥测键，以便不同网关的数据能在平台中统一查看。
- 作为运维人员，我希望 JSON 格式错误或字段缺失时有明确的错误信息和原始数据上下文，以便快速定位问题。
- 作为平台管理人员，我希望边缘网关的批量上传能自动识别或创建子设备，以便减少手工配置工作量。
- 作为后续模块使用方，我希望边缘 JSON 接入的数据与透传网关采集的数据走相同的遥测存储链路，以便统一查询和统计。

## 6. 需求

### 6.1 功能需求

- 系统必须支持边缘网关通过 HTTP POST 上传单设备 JSON 遥测数据。
- 系统必须支持边缘网关通过 MQTT 上传单设备 JSON 遥测数据。
- 系统必须支持边缘网关通过 HTTP/MQTT 上传批量设备 JSON 遥测数据（ThingsBoard 兼容格式）。
- 系统必须支持边缘网关上传设备属性（Attributes），区分遥测和属性。
- 系统必须校验 JSON 格式的正确性（合法 JSON、必填字段存在性）。
- 系统必须校验上传网关的身份（AccessToken / DevicePassword）。
- 系统必须支持 JSON 字段名到平台遥测键的 1:1 映射配置。
- 系统必须支持数值型、布尔型、字符串型和嵌套 JSON 型遥测数据。
- 上传成功后数据必须写入 `TelemetryLatest` 和 `TelemetryData` 存储链路。
- 上传失败时必须记录错误日志，包含网关标识、设备标识、错误原因和原始数据上下文（截断后）。
- 网关批量上传中子设备不存在时应自动创建（继承网关租户和客户归属）。
- 批量上传中单个子设备失败不应影响其他子设备的数据处理。
- 系统必须支持边缘网关上传子设备 connect/disconnect 状态。

### 6.2 当前已具备能力

- `TelemetryController`（MQTT）已支持 JSON/XML/Binary 遥测上传。
- `AttributesController`（MQTT）已支持属性上报和请求。
- `GatewayController`（MQTT）已支持 ThingsBoard 兼容网关批量遥测、属性、connect/disconnect、KepServerEx 数据入口。
- `DevicesController` 已提供 HTTP 遥测上传端点。
- CAP EventBus 遥测发布链路已工作。
- `JudgeOrCreateNewDevice` 已支持网关子设备自动创建。

### 6.3 后续补齐需求

- 定义边缘网关 JSON 上传的标准 payload 格式文档（单设备、批量）。
- 完善 `ProduceDataMapping` 或等效的字段映射配置，明确映射的生效范围（全局 / 产品级 / 设备级）。
- 增强 JSON 解析失败时的错误日志，包含设备名和截断后的原始数据。
- 统一 HTTP 和 MQTT 两侧的 JSON 校验逻辑，避免行为分裂。
- 增加网关批量上传的 HTTP 端点（当前主要在 MQTT 侧实现）。
- 增加边缘 JSON 接入的最小测试用例，覆盖格式校验、字段映射、批量上传和错误日志。

### 6.4 非功能需求

- JSON 解析不得阻塞 MQTT 或 HTTP 接入线程。
- 批量上传中单个设备处理异常不得影响其他设备。
- 错误日志必须包含足够的排障信息，但原始数据应截断避免日志膨胀。
- 字段映射配置的变更应即时生效或在合理时间内生效。
- 边缘 JSON 接入不得绕过设备租户和客户隔离。
- 接入数据必须复用现有遥测发布链路，不得另建存储路径。

## 7. 验收标准

### 7.1 单设备 JSON 上传

- Given 边缘网关持有有效 AccessToken
  When 通过 HTTP POST JSON 到 `/api/devices/{devname}/telemetry`
  Then 系统应校验 JSON 格式，成功后数据写入遥测链路。

- Given 边缘网关通过 MQTT 连接
  When 向 `devices/{devname}/telemetry` 发布 JSON payload
  Then 系统应解析并写入遥测链路。

### 7.2 批量设备 JSON 上传

- Given 边缘网关已连接 MQTT
  When 向 `v1/gateway/telemetry` 发布 ThingsBoard 兼容 JSON
  Then 系统应为每个子设备分别写入遥测数据，不存在的子设备自动创建。

- Given 批量 JSON 中某个子设备数据格式错误
  When 系统处理批量上传
  Then 该子设备处理失败应记录日志，其他子设备正常处理不受影响。

### 7.3 字段映射

- Given 边缘网关上传 JSON `{"temp": 24.5}`
  When 配置了字段映射 `temp → temperature`
  Then 系统应以遥测键 `temperature` 存储值 `24.5`。

### 7.4 错误诊断

- Given 边缘网关上传非法 JSON
  When 系统解析失败
  Then 日志应包含网关标识、设备标识、错误原因和截断后的原始数据。

### 7.5 属性上传

- Given 边缘网关上传属性数据
  When 系统处理属性上传
  Then 属性值应更新到 `AttributeLatest`，与遥测数据区分存储。

## 8. 不在范围内

- 不在本模块实现 JSON Schema 版本管理和远程推送。
- 不在本模块实现复杂字段映射表达式（嵌套提取、正则替换、脚本转换）。
- 不在本模块实现边缘网关 JSON 格式的远程配置下发。
- 不在本模块实现字段映射失败的详细诊断界面。
- 不在本模块处理透传网关 Modbus 采集。

## 9. 依赖与后续关系

- 依赖"网关与设备接入模块"提供网关认证和子设备管理能力。
- 依赖遥测存储链路（CAP EventBus + TimescaleDB）。
- 本模块是"点位、模板与映射模块"的数据入口之一。
- 本模块是"实时数据与历史数据查询模块"的数据来源之一。

## 10. 当前状态判断

**状态：基础能力已有，需补齐格式规范、字段映射和诊断能力。**

MQTT 侧的 ThingsBoard 兼容网关协议已工作，HTTP 侧单设备上传已工作。后续应以标准化 JSON 格式、统一校验逻辑、完善字段映射和增强错误诊断为主，不涉及大规模重构。
