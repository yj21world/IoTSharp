## 1. MQTT Broker 的横向扩展问题（§6.2）

**决策：单实例部署，扩展问题推迟。**

代码探查确认：MQTTnet 内置 Broker 的会话和订阅状态保存在进程内存中（`WithPersistentSessions()` 为内存级，无数据库/Redis 后端）。第一阶段只做小项目、少量设备、单实例部署，进程重启导致会话丢失是可接受的。横向扩展问题推迟到后续阶段。

## 2. Modbus 采集调度的时序与并发控制（§5.2）

**决策：维持现有实现。**

代码确认现有机制已满足需求：
- 同一网关下的请求通过 `ConcurrentDictionary<string, PendingRequest>` 保证串行（一次只允许一个飞测请求）。
- 采集调度通过 `GatewayScheduler` 的三级优先级队列管理：高速（<15s）→ 10s 检查，中速（15-45s）→ 30s，低速（>45s）→ 60s。
- 调度方式为 `while` 循环 + `Task.Delay(100ms)`，不是 Quartz Job。
- 超时默认 5000ms，超时后记录日志并移除待处理请求，不自动即时重试。`CollectionConnectionDto.RetryCount` 字段已定义但未被使用，当前依赖调度周期的自然重试。即时重试推迟到后续阶段。

## 3. 控制命令状态机的具体设计（§5.5）

**决策：第一阶段做最简版 RPC 下发。**

当前代码已有 RPC 通道（`RpcClient`、`RpcController`），支持基于 MQTT 的请求/响应模式。第一阶段沿用现有通道，补齐操作日志记录和基本权限校验。完整状态机（Pending → Sent → Executed → Confirmed / Failed / Timeout）推迟到后续阶段实现。

## 4. 业务设备建模的边界（§5.3）

**决策：第一阶段暂不考虑。**

先跑通通信层闭环（网关注册 → 采集 → 遥测 → 控制），业务设备聚合建模后续再做。

## 5. 时序数据写入策略（§5.4）

**决策：TimescaleDB，全量写入，不做保留策略。**

- 时序存储选用 TimescaleDB（与 PostgreSQL 统一运维，已在 docker-compose 中使用 `timescaledb-ha:pg17`）。
- 所有遥测同时写 `TelemetryData`（历史超表）和 `TelemetryLatest`（PostgreSQL 最新值），不做点位级区分。
- 不做降采样，不设自动数据保留清理。后续根据运维情况再调整。
- 数据量估算（10 网关 × 50 点位 × 30s ≈ 216 万条/天）完全在 TimescaleDB 能力范围内。

## 6. 事件总线的使用边界（§3.2）

**决策：保持 CAP + RabbitMQ 架构不变。**

- 遥测数据逐条发布 CAP 事件（topic: `iotsharp.services.datastream.telemetrydata`），不做批量聚合。
- 对于 10 个网关的规模，消息吞吐完全在 RabbitMQ 和 CAP 能力范围内。
- CAP 自带消息持久化和重试机制（失败重试、成功消息 6 小时后清理），无需额外设计幂等。
- 虽然第一阶段不启用规则链，但保持 CAP 架构不变可避免后续重新接入的成本。

## 7. 规则链引擎的适用范围（§5.6）

**决策：第一阶段暂不考虑。**

## 8. 前端架构的缺失（§4.3）

`ClientApp` 为 Vue 3 + Element Plus。第一阶段只提供基础的设备管理、数据查看页面，不做组态、仪表盘和高级操作面板。

当前项目规模和客户数量不需要在多租户隔离上投入，保留现有 `IJustMy` 查询约束即可。
