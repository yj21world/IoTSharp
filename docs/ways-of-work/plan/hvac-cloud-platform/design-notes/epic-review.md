## 3. 模块分类（按架构角色）

### A. 接入层（2个模块）
- **M1 网关与设备接入** — 身份认证、Topic路由、在线状态、子设备管理
- **M3 边缘网关 JSON 接入** — JSON格式定义、版本管理、字段映射、校验诊断

### B. 采集层（2个模块）
- **M2 透传网关 Modbus 采集** — 云端调度、请求生成、响应解析、日志闭环
- **M4 点位、模板与映射** — 设备类型模板、采集点模板、控制点模板、协议到遥测的映射

### C. 业务层（2个模块）
- **M5 暖通业务设备建模** — 底层通信设备与业务设备分离、多数据源聚合、设备分类视图
- **M6 控制命令与审计** — 命令模板、状态机、超时处理、权限校验、审计追踪

### D. 数据与统计层（3个模块）
- **M7 实时与历史数据查询** — 最新值查询、时间序列查询、业务设备视图查询
- **M8 能耗统计** — 小时/日/月/年统计、站点级/系统级/设备级对比
- **M9 告警与规则链暖通化** — 通信告警、阈值告警、控制失败告警、规则链触发

### E. 平台管理层（1个模块）
- **M10 权限、菜单与租户管理** — 租户隔离、角色权限、菜单导航、操作审计

## 4. 模块间依赖关系

```
M1 网关接入
  ├──→ M2 透传 Modbus 采集（需要网关在线状态、租户归属）
  ├──→ M3 边缘 JSON 接入（需要网关身份认证、Topic路由）
  └──→ M6 控制命令（需要MQTT通道、设备身份）

M4 点位模板
  ├──→ M2 透传 Modbus 采集（需要点位配置、转换规则）
  └──→ M3 边缘 JSON 接入（需要字段映射定义）

M2 + M3（采集数据入库）
  └──→ M5 业务设备建模（需要遥测数据源）
  └──→ M7 实时/历史查询（需要入库数据）

M5 业务设备建模
  └──→ M6 控制命令（控制业务设备而非通信设备）
  └──→ M8 能耗统计（需要业务设备视图）

M6 控制命令
  └──→ M9 告警规则（控制失败触发告警）

M7 + M8（数据与统计）
  └──→ M9 告警规则（阈值告警需要实时/统计数据）
```

## 5. 建议的下一步行动

**优先级最高（立即可做）：**
1. **M4 点位、模板与映射模块** — M2 和 M3 的共同前置依赖。PRD 应定义：设备类型模板结构、采集点模板、控制点模板、点位到遥测的映射规则、模板继承和复用机制。
2. **M6 控制命令与审计模块** — 阶段一闭环的关键一环。PRD 应定义：命令类型、状态机、超时检测机制、权限和审计要求、透传网关 vs 边缘网关的命令下发差异。
3. **M3 边缘网关 JSON 接入模块** — PRD 应定义：边缘网关上传的 JSON 标准格式（含版本号）、字段白名单、云端映射规则、格式错误处理、版本兼容策略、诊断日志。

**建议顺序：** M4 → M3 → M6 → M5 → M7 → M8 → M9 → M10
