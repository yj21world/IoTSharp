# Feature PRD：平台权限、菜单与租户管理完善模块

## 1. Feature 名称

平台权限、菜单与租户管理完善模块

## 2. 所属 Epic

- 父级产品需求：[暖通空调云端物联网平台长期产品需求](../epic.md)
- 父级架构文档：[暖通空调云端物联网平台架构规格](../arch.md)

## 3. 目标

### 问题

当前系统已具备 ASP.NET Core Identity + JWT Bearer 用户认证、`TenantId`/`CustomerId` JWT Claims 和 `IJustMy` 查询约束等基础多租户能力。但角色权限体系、菜单管理和租户管理功能尚不完整。对于多项目、多客户的暖通运维场景，需要完善的租户-客户-用户-角色-权限体系和动态菜单管理能力。这不是暖通业务特有的需求，而是 SaaS 平台通用的基础能力。

### 解决方案

在现有 Identity 基础设施上，完善租户管理（创建、配置、启用/禁用）、角色与权限分配（RBAC）、动态菜单管理和操作审计日志。第一阶段保留现有基础能力不变，待暖通核心业务闭环稳定后再全面完善平台管理功能。

### 影响

完善的平台管理能力使系统能够同时服务多个客户和项目，降低运维成本和部署复杂度。多租户隔离是后续 SaaS 化运营的前提。

## 4. 使用场景

- 平台管理员创建新租户和客户。
- 平台管理员为不同角色分配设备和数据权限。
- 调试工程师只能查看和操作自己负责的站点设备。
- 菜单根据用户角色动态展示（管理员看到全部菜单，运维人员只看到设备和数据菜单）。
- 审计人员查询用户操作记录（登录、数据修改、控制命令）。

## 5. 用户故事

- 作为平台管理员，我希望能创建和管理租户，以便为不同客户提供隔离的平台服务。
- 作为平台管理员，我希望能定义角色并分配菜单和数据权限，以便控制不同用户的访问范围。
- 作为运维人员，我希望能只看到我有权限的设备和站点，以避免信息过载和误操作。
- 作为审计人员，我希望能查询用户的操作记录，以便追溯问题。

## 6. 需求

### 6.1 功能需求（目标态）

- 系统必须支持租户创建、配置、启用/禁用。
- 系统必须支持客户（Customer）与租户的层级关系管理。
- 系统必须支持角色定义与权限分配（RBAC）。
- 系统必须支持操作权限与数据权限的分离。
- 系统必须支持前端菜单的动态配置和权限控制。
- 系统必须支持按角色显示/隐藏菜单项。
- 系统必须支持操作审计日志的查询和导出。
- 多租户数据隔离采用逻辑隔离（共享数据库，通过 `TenantId` 过滤）。
- 新增/调整 API 必须遵循 `ApiResult<T>`。

### 6.2 当前已具备能力

- ASP.NET Core Identity — 用户、角色、Claims。
- JWT Bearer 认证 — 已配置 `TenantId`、`CustomerId` Claims。
- `IJustMy` 查询约束 — 租户/客户级别数据过滤。
- `AuditLog` 实体 — 操作审计。

### 6.3 待确认问题

- 多租户隔离模式：逻辑隔离（共享数据库 + TenantId 过滤）还是物理隔离（单租户单库）？当前建议逻辑隔离。
- 菜单权限是前端路由控制还是需要后端 API 级别校验？建议两者都需要。
- 是否需要数据权限的细粒度控制（如"只能看自己负责的站点"）？第一阶段可以不实现。
- 站点、项目、客户之间的层级关系需要明确定义。

### 6.4 非功能需求（目标态）

- 权限校验不得显著增加 API 响应时间。
- 菜单权限的变更应立即生效（或清除缓存后生效）。
- 审计日志不得包含用户密码、Token 等敏感信息。

## 7. 验收标准

待第四阶段详细设计时定义。

## 8. 不在范围内（第一阶段）

- 不在第一阶段完整重做租户、角色、菜单和权限系统。
- 不在第一阶段实现数据权限的细粒度控制。
- 不在第一阶段实现操作审计日志的图形化查询界面。

## 9. 第一阶段可做的准备

- 保持现有 `TenantId`、`CustomerId` JWT Claims 和 `IJustMy` 查询机制不变。
- 新增实体时继续携带租户/客户归属字段。
- API 端点继续保持 `[Authorize]` 标记。
- 收集真实使用场景中的权限需求，为后续设计提供输入。

## 10. 当前状态判断

**状态：第四阶段模块，最后处理。现有 Identity 基础设施可支撑第一阶段核心业务闭环。**

当前 ASP.NET Core Identity + JWT + `IJustMy` 已满足第一阶段单项目/少量用户场景的权限隔离需求。完整的平台管理功能是 SaaS 化的前提，但不是暖通数据与控制闭环的前提，优先级排在最后。
