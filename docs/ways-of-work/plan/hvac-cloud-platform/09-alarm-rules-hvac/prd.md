# Feature PRD：告警与规则链暖通化模块

## 1. Feature 名称

告警与规则链暖通化模块

## 2. 所属 Epic

- 父级产品需求：[暖通空调云端物联网平台长期产品需求](../epic.md)
- 父级架构文档：[暖通空调云端物联网平台架构规格](../arch.md)
- 前置模块：[实时数据与历史数据查询模块](../06-telemetry-data-query/prd.md)、[控制命令与审计模块](../05-control-command-audit/prd.md)

## 3. 目标

### 问题

暖通空调系统运维中，通信异常（网关离线、采集超时）、设备故障（过载、过热、频率异常）、参数越限（温度过高、压力过低）和控制失败等事件需要被及时发现和处理。当前系统已具备 `FlowRule`、`FlowRuleProcessor`、`Alarm` 和 `TaskActions` 等规则链基础设施，但目前主要面向通用物联网场景，缺少针对暖通空调领域的告警规则、时间窗口聚合判断和控制联动能力。

### 解决方案

在现有规则链引擎基础之上，围绕暖通场景做领域化增强。定义暖通常见的告警类型（通信告警、阈值告警、异常值告警、控制失败告警），支持基于遥测事件的规则触发、告警生成、告警生命周期管理（触发 → 确认 → 清除）。后续阶段支持规则链与控制命令联动，形成自动化的告警-控制闭环。

### 影响

告警能力使平台从"被动查看数据"升级为"主动发现问题"。通信异常和阈值越限的及时发现可以避免设备损坏和能耗浪费。

## 4. 使用场景

- 运维人员收到告警："1#水泵 频率超过 50Hz 已持续 3 分钟"。
- 网关离线超过 5 分钟后自动触发通信告警。
- 采集 CRC 错误率超过阈值时触发告警，提醒现场检查通信线路。
- 规则触发后自动向相关人员发送通知（后续阶段）。
- 告警确认和清除——运维人员处理完故障后在平台中标记告警为已处理。

## 5. 用户故事

- 作为运维人员，我希望能收到设备参数越限告警（如温度过高、压力过低），以便及时处理异常。
- 作为运维人员，我希望告警不是瞬时触发而是需要持续条件满足才告警（如连续 3 次超限），以避免误报。
- 作为调试人员，我希望能收到通信异常告警（网关离线、采集超时率过高），以便排查现场网络问题。
- 作为系统管理员，我希望能管理告警的确认、清除和归档生命周期，以便跟踪处理进度。
- 作为平台管理人员，我希望后续能配置规则链在触发告警的同时自动下发控制命令（如温度过高时自动启动冷却泵）。

## 6. 需求

### 6.1 功能需求（目标态）

- 系统必须支持定义暖通告警规则（通信告警、阈值告警、异常值告警、控制失败告警）。
- 系统必须支持基于遥测事件的规则触发（由 CAP 事件驱动）。
- 系统必须支持时间窗口聚合判断（如"连续 N 次超限"、"持续 M 分钟"）。
- 系统必须支持告警生命周期管理（触发 → 确认 → 清除 → 归档）。
- 系统必须支持告警列表查询，按设备、类型、状态、时间范围过滤。
- 系统保留告警与规则链联动控制命令的扩展点（后续阶段）。
- 告警消息应关联到具体设备、遥测键、触发值和阈值。

### 6.2 当前已具备能力

- `FlowRule` / `Flow` / `FlowOperation` 实体 — 规则链定义。
- `FlowRuleProcessor` — 规则评估引擎，通过 `RunRules` 委托在遥测入库事件中触发。
- `Alarm` / `DeviceRule` 实体 — 告警和规则定义。
- `TaskActions` — 规则链动作（如 `PublishTelemetryDataTask`）。
- CAP EventBus — 规则链触发的事件源。

### 6.3 待确认问题

- 规则链是否需要图形化编排界面（类似 Node-RED）？第一阶段可以仅用配置式规则。
- 告警通知渠道的范围（平台内消息、邮件、短信、Webhook）。
- 告警自动升级策略（如未确认告警超过 N 分钟后自动升级通知级别）。

### 6.4 非功能需求（目标态）

- 告警判断不得阻塞遥测入库主链路。
- 告警规则需考虑高频采集场景下的性能（如每秒数千条遥测时避免逐条触发规则评估）。
- 告警历史数据需有保留和归档策略。

## 7. 验收标准

待第三阶段详细设计时定义。

## 8. 不在范围内（第一阶段）

- 不在第一阶段实现暖通告警规则和规则链配置。
- 不在第一阶段实现告警通知（邮件、短信等）。
- 不在第一阶段实现告警-控制联动闭环。
- 不在第一阶段修改规则链引擎核心逻辑。

## 9. 第一阶段可做的准备

- 保持 `FlowRuleProcessor.RunRules` 调用点在遥测入库链路中不变。
- 保留 `FlowRule` / `Alarm` 实体和数据库表结构。
- 在采集侧预留简单的采集状态标记（成功/失败计数），为后续通信告警提供数据基础。

## 10. 当前状态判断

**状态：第三阶段模块，第一阶段暂不实施。规则链基础设施已存在，后续重点是暖通领域化规则和告警管理。**

现有规则链引擎已能满足基础的规则评估需求。后续暖通化增强应围绕告警类型定义、时间窗口聚合、告警生命周期管理和通知渠道展开，不需要重写规则引擎核心。
