# Implementation Plan：告警与规则链暖通化模块

> 对应 PRD：[告警与规则链暖通化模块 PRD](./prd.md)  
> 审核日期：2026-04-28  
> 状态：**通用规则/告警已有，暖通模板化待实施**

## 1. 实施概述

当前仓库已有 `Alarm`、`AlarmController`、`FlowRule`、`Flow`、`FlowOperation`、`FlowRuleProcessor` 和前端规则页面。第一阶段不重写规则引擎，而是围绕暖通常见场景补告警模板、事件来源、控制失败告警和通信异常告警。

## 2. 依赖前置

- 采集日志和遥测链路能稳定产生数据。
- 控制命令模块能记录 `CommandLog` 状态。
- 点位模板能标识阈值、单位、告警级别和启用状态。

## 3. 涉及文件

| 优先级 | 文件 | 动作 |
|--------|------|------|
| P0 | `IoTSharp.Data/Alarm.cs`、`IoTSharp/Controllers/AlarmController.cs` | 审查暖通告警字段是否足够 |
| P0 | `IoTSharp/FlowRuleEngine/*` | 复用现有规则链触发，不重写 |
| P1 | `IoTSharp.Data/CollectionLog.cs` | 将采集失败/超时转换为通信告警来源 |
| P1 | `IoTSharp.Data/CommandLog.cs` | 控制失败/超时生成告警，依赖 05 完成 |
| P1 | `ClientApp/src/views/iot/alarms/*`、`rules/*` | 增加暖通告警模板和列表过滤 |
| P2 | `IoTSharp.Test/` | 增加阈值、通信失败、控制失败测试 |

## 4. 实施步骤

1. 明确告警来源：遥测阈值、采集超时、网关离线、控制失败。
2. 定义暖通告警模板 DTO：设备类型、点位 key、条件、级别、恢复条件、去抖时间。
3. 优先使用现有 `FlowRule` 表达阈值逻辑，必要时新增模板到规则链的生成服务。
4. 采集超时和控制失败不走复杂脚本，直接在对应服务处生成标准告警事件。
5. 前端在告警列表增加设备类型、业务设备、来源、级别、确认/清除过滤。

## 5. 验收标准

- 遥测超限可生成告警，并支持确认/清除。
- 采集连续超时可生成通信告警。
- 控制失败或超时可生成控制告警。
- 告警查询遵循租户/客户隔离和统一返回结构。
