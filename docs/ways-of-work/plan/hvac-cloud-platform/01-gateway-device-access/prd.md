# Feature PRD：网关与设备接入模块

## 1. Feature 名称

网关与设备接入模块

## 2. 所属 Epic

- 父级产品需求：[暖通空调云端物联网平台长期产品需求](../epic.md)
- 父级架构文档：[暖通空调云端物联网平台架构规格](../arch.md)

## 3. 目标

### 问题

暖通空调云端平台必须先具备稳定的设备和网关接入能力，后续的 Modbus 采集、边缘 JSON 上传、实时数据、历史数据、控制下发和能耗统计才能成立。当前仓库已经具备较完整的接入基础，包括 HTTP 设备 API、内置 MQTT Broker、MQTT 设备/网关 Topic 路由、设备身份认证、网关子设备自动创建、连接/活动状态更新、遥测/属性/告警/RPC 通道等能力。但这些能力需要形成明确的产品需求边界和验收标准，避免后续模块重复实现接入逻辑或破坏现有路径。

### 解决方案

将现有接入能力定义为平台基础模块：以 `Device`、`Gateway`、`DeviceIdentity`、`Tenant`、`Customer` 为核心模型，支持设备和网关通过 HTTP、MQTT 接入云端，并记录连接状态、活动状态、最后活动时间和网关子设备关系。当前状态数据通过服务侧属性写入 `AttributeLatest`，不是 `Device` 表上的独立在线字段。该模块不负责复杂 Modbus 采集、业务设备聚合、能耗统计和组态展示，只提供可靠的接入、身份、路由、状态和基础数据入口。

### 影响

该模块作为“已基本完成、后续补齐约束”的基础能力，应为所有后续模块提供统一接入入口。后续透传网关采集、边缘 JSON 接入、点位映射、控制命令和设备建模，都应复用该模块的设备身份、网关关系、MQTT/HTTP 通道和在线状态能力。

## 4. 使用场景

长期需求文档不重点展开用户角色，本模块仅按使用场景描述：

- 平台管理人员创建设备、网关和认证信息。
- 现场调试人员让网关接入云端，并确认在线状态。
- 网关通过 MQTT 上传子设备连接状态、遥测、属性、告警或原始数据。
- 普通设备通过 HTTP 或 MQTT 上传数据。
- 后续控制模块通过已有接入通道向设备或网关下发命令。

## 5. 用户故事

- 作为平台管理人员，我希望能创建设备和网关，并配置认证方式，以便现场设备可以安全接入云端。
- 作为现场调试人员，我希望网关上线后平台能显示在线状态和最后活动时间，以便判断现场通信是否正常。
- 作为现场调试人员，我希望网关上报子设备数据时平台能自动识别或创建子设备，以便减少现场批量配置工作量。
- 作为平台接入模块使用方，我希望 HTTP 和 MQTT 数据入口遵循统一设备身份和租户归属，以便后续采集、控制、告警模块不重复实现认证逻辑。
- 作为后续控制模块，我希望能够通过已有 MQTT/RPC 通道定位设备或网关，以便下发控制命令并接收执行反馈。
- 作为系统维护人员，我希望接入失败、认证失败、网关数据格式异常等问题有日志和可查询状态，以便排查现场问题。

## 6. 需求

### 6.1 功能需求

- 系统必须支持创建设备和网关两类接入对象，设备类型使用现有 `DeviceType.Device` 与 `DeviceType.Gateway`。
- 系统必须为每个设备或网关维护 `DeviceIdentity`，支持 `AccessToken`、`DevicePassword`、`X509Certificate` 三类认证方式。
- 系统必须支持通过 HTTP API 创建设备、查询设备列表、查询设备详情、更新设备、删除设备和查询认证信息。
- 系统必须支持通过 HTTP 接收设备遥测、属性、告警和原始数据。
- 系统必须支持通过 MQTT 接收设备遥测、属性、告警、RPC 和网关数据。
- 系统必须支持内置 MQTT Broker，并通过 MQTT Controller 路由设备和网关 Topic。
- 系统必须支持网关作为父设备，普通设备作为网关子设备。
- 当网关上报子设备名称时，系统应能基于网关租户和客户归属自动创建或查找子设备。
- 系统必须记录设备和网关的在线/离线状态、活动状态和最后活动时间；当前实现以 `_Connected`、`_Active`、`_LastActivityDateTime`、`_LastConnectDateTime` 等服务侧属性承载。
- 系统必须支持网关上报子设备 connect/disconnect 状态。
- 系统必须支持 ThingsBoard 风格的 `v1/gateway` 网关兼容路由。
- 系统必须将接入数据发布到事件总线或遥测处理链路，供实时数据、历史数据、告警和规则链使用。
- 系统必须保持租户和客户隔离，设备查询和操作不得越权访问其他租户或客户数据。
- 系统必须在 API 返回中遵循统一 `ApiResult<T>` 结构；列表和查询类结果应使用 `{ total, rows }`。
- 系统必须保留 `HVACDeviceType` 字段作为暖通设备分类入口，但本模块不负责完整业务设备聚合。
- 系统应允许设备绑定产品或设备类型模板，为后续点位、采集和控制模块提供配置来源。

### 6.2 当前已具备能力

- `Device` 与 `Gateway` 模型已经存在，`Gateway` 继承自 `Device`。
- `DeviceIdentity` 已支持 `AccessToken`、`DevicePassword`、`X509Certificate`。
- `DevicesController` 已提供设备管理、设备身份、遥测、属性、告警、RPC、原始数据上传等 HTTP 接口。
- `MQTTController` 体系已提供遥测、属性、告警、RPC、网关数据等 MQTT 接入。
- `GatewayController` 已支持网关遥测、属性、connect/disconnect、JSON/XML 原始数据和 KepServerEx 数据入口。
- `JudgeOrCreateNewDevice` 已支持网关子设备自动创建。
- 连接状态和活动状态已通过 CAP 事件写入服务侧属性，设备列表和详情可读取活动状态与最后活动时间。
- `TelemetryLatest`、`AttributeLatest`、事件总线和遥测存储链路已可承接接入数据。

### 6.3 后续补齐需求

- 梳理并统一设备管理 API 的返回结构，逐步消除列表接口返回形态不一致的问题。
- 将设备/网关前端页面从 `fs-crud` / `@fast-crud/fast-crud` 迁移为 Vue 3 + Element Plus 显式组件实现，满足当前前端约束。
- 对网关类型进行更明确的产品表达，区分透传网关、边缘采集网关和普通 MQTT 网关。
- 增加网关接入诊断信息，例如最近认证失败原因、最近数据格式错误、最近心跳时间和最近 Topic。
- 明确自动创建子设备的命名冲突规则，避免同一网关下重复或误创建子设备。
- 明确网关子设备是否允许手工绑定、解绑和重命名。
- 为 HTTP 和 MQTT 接入补齐面向暖通项目的示例 payload 文档。
- 为接入模块补齐最小化测试用例，包括创建设备、创建网关、MQTT 上线、子设备自动创建、HTTP 遥测上传和租户隔离。

### 6.4 非功能需求

- 接入模块必须保持向后兼容，不得破坏现有 HTTP/MQTT 设备接入路径。
- 接入模块不应引入新的消息中间件或关系数据库类型，继续使用 PostgreSQL、CAP、RabbitMQ 和现有遥测存储。
- MQTT 接入处理应避免长时间阻塞，复杂采集、解析和统计逻辑必须交给后续模块或后台服务。
- 接入日志应能支持现场调试，不应只记录不可定位的通用异常。
- 设备认证信息返回时必须避免泄露敏感值，尤其是 X509 私钥、设备密码等。
- 自动创建子设备必须继承父网关的租户和客户归属。
- 模块设计必须兼容后续多实例部署，但当前阶段不强制完成 MQTT Broker 集群化。

## 7. 验收标准

### 7.1 设备与网关管理

- Given 已登录用户具有设备管理权限  
  When 创建普通设备  
  Then 系统应创建 `DeviceType.Device` 设备和对应 `DeviceIdentity`。

- Given 已登录用户具有设备管理权限  
  When 创建网关  
  Then 系统应创建 `DeviceType.Gateway` 设备和对应 `DeviceIdentity`。

- Given 用户查询设备列表  
  When 指定客户和分页参数  
  Then 系统应返回符合租户/客户归属的数据，并包含 `total` 与 `rows`。

### 7.2 设备认证

- Given 设备使用 AccessToken  
  When 通过 HTTP 或 MQTT 上报数据  
  Then 系统应能识别对应设备并写入活动状态。

- Given 设备使用 DevicePassword  
  When 设备连接 MQTT Broker  
  Then 系统应校验设备身份和密码。

- Given 设备使用 X509Certificate  
  When 用户查询设备身份  
  Then API 不应直接暴露私钥等敏感内容。

### 7.3 MQTT 网关接入

- Given 一个网关设备已存在并成功连接 MQTT  
  When 网关向 `gateway/{devname}/connect` 或等效路由上报子设备上线  
  Then 系统应创建或查找该子设备，并记录子设备在线状态。

- Given 网关上报子设备遥测数据  
  When 子设备不存在  
  Then 系统应在网关租户和客户下自动创建子设备，并写入遥测数据。

- Given 网关上报子设备遥测数据  
  When 子设备已存在  
  Then 系统应复用已有子设备，不得重复创建。

### 7.4 HTTP 数据接入

- Given 一个设备 AccessToken 有效  
  When 设备调用 HTTP 遥测上传接口  
  Then 系统应写入最新值和历史遥测处理链路。

- Given 一个设备 AccessToken 无效  
  When 设备调用 HTTP 数据入口  
  Then 系统应拒绝请求，并返回统一错误结构。

### 7.5 状态与诊断

- Given 设备或网关成功上传数据  
  When 查询设备列表或详情  
  Then 应能看到活动状态和最后活动时间。

- Given 网关上传格式错误数据  
  When 系统无法解析 payload  
  Then 应记录错误日志，并避免影响其他设备数据处理。

## 8. 不在范围内

- 不在本模块实现云端 Modbus 采集调度、请求生成和响应解析。
- 不在本模块实现边缘 JSON 字段映射规则和版本治理，相关内容归入边缘网关 JSON 接入模块。
- 不在本模块实现暖通业务设备聚合，例如水泵聚合变频器、电表和传感器。
- 不在本模块实现能耗统计。
- 不在本模块重做租户、角色、菜单和权限系统。
- 不在本模块实现完整组态功能。
- 不在本模块实现 MQTT Broker 集群化或网关接入层独立服务化。
- 不在本模块彻底清理所有历史 API，仅对影响接入主路径的接口进行整理。

## 9. 依赖与后续关系

- 本模块是“透传网关 Modbus 采集模块”的前置基础。
- 本模块是“边缘网关 JSON 接入模块”的前置基础。
- 本模块是“控制命令与审计模块”的通道基础。
- 本模块是“实时数据与历史数据查询模块”的数据入口基础。
- 本模块是“暖通业务设备建模模块”的底层设备来源。

## 10. 当前状态判断

当前代码已经具备本模块的大部分基础能力，不建议将其作为大规模重构任务处理。后续实施应以小步补齐为主，优先完成 API 返回一致性、网关类型表达、接入诊断信息、Payload 示例和基础测试。
