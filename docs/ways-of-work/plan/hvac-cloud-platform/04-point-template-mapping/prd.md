# Feature PRD：点位、模板与映射模块

## 1. Feature 名称

点位、模板与映射模块

## 2. 所属 Epic

- 父级产品需求：[暖通空调云端物联网平台长期产品需求](../epic.md)
- 父级架构文档：[暖通空调云端物联网平台架构规格](../arch.md)
- 前置模块：[网关与设备接入模块](../01-gateway-device-access/prd.md)、[透传网关 Modbus 采集模块](../02-transparent-modbus-collection/prd.md)

## 3. 目标

### 问题

在暖通项目中，同类型设备（如西门子 V20 变频器）的参数配置（功能码、寄存器地址、数据类型、倍率、单位）在不同项目中高度重复。如果没有设备类型模板机制，调试工程师需要为每台设备手工录入十几个甚至几十个点位配置，效率低且容易出错。此外，不同边缘网关厂商的 JSON 字段命名风格不同（如 `temp`、`temperature`、`TEMP_VALUE`），需要一种机制将外部字段统一映射到平台遥测键。

### 解决方案

在现有 `DeviceTypeProfile`、`CollectionRuleTemplate`、`ProduceDataMapping` 实体基础上，建立可复用的设备类型模板体系。模板中可预定义遥测点位（含完整 Modbus 参数）和控制点位。模板应用到设备时自动生成 `CollectionPoint` 记录，并支持覆盖个别参数。同时提供数据映射规则，支持 JSON 字段名到遥测键的 1:1 映射。

### 影响

模板机制大幅减少重复配置工作，提高现场调试效率。统一的点位定义也为后续控制命令模板、能耗统计点位标记和告警规则引用提供了基础数据。

## 4. 使用场景

- 调试工程师创建"西门子 V20 变频器"设备类型模板，定义常用点位。
- 在创建采集设备时选择模板，自动填充所有点位配置。
- 根据现场实际微调个别点位参数（如从站地址），不影响模板定义。
- 配置边缘网关 JSON 字段映射规则，使不同网关的数据统一到平台遥测键。
- 产品（Produce）与设备类型模板关联，选择产品即获得默认点位配置。

## 5. 用户故事

- 作为调试工程师，我希望能创建暖通设备类型模板并预定义点位，以便同类型设备能快速复用配置。
- 作为调试工程师，我希望能将模板应用到设备时自动生成采集点位，以便减少手工录入。
- 作为调试工程师，我希望模板应用后仍能覆盖个别点位参数，以便适配现场差异。
- 作为现场调试人员，我希望能配置 JSON 字段到遥测键的映射规则，以便不同网关的数据能统一。
- 作为平台管理人员，我希望模板的修改不影响已应用设备，或提供明确的更新机制。

## 6. 需求

### 6.1 功能需求

- 系统必须支持创建/编辑/删除设备类型模板（`DeviceTypeProfile`）。
- 模板必须支持预定义遥测点位（键名、显示名、单位、数据类型）。
- 模板必须支持预定义 Modbus 通信参数（从站地址、功能码、寄存器地址、数量、字节序）。
- 模板必须支持预定义变换管道（scale / offset / clamp）。
- 模板必须支持预定义属性点位和控制点位。
- 系统必须支持将模板应用到设备，自动生成对应的 `CollectionPoint` 记录。
- 模板应用时已有同名点位应可配置为跳过或覆盖。
- 系统必须支持创建/编辑/删除采集规则模板（`CollectionRuleTemplate`）。
- 规则模板应可被多个采集任务引用。
- 系统必须支持创建/编辑/删除数据映射规则（`ProduceDataMapping`）。
- 映射规则必须支持 JSON 字段名到平台遥测键的 1:1 映射。
- 产品（`Produce`）必须支持关联设备类型模板。
- 新增/调整 API 必须遵循 `ApiResult<T>`，列表/查询使用 `{ total, rows }`。

### 6.2 当前已具备能力

- `DeviceTypeProfile`、`CollectionRuleTemplate`、`ProduceDataMapping` 实体已存在。
- `DeviceTypeProfileService` 已存在，需确认模板应用逻辑完整性。
- `Produce` 产品模型已存在，可关联模板。

### 6.3 后续补齐需求

- 审查 `DeviceTypeProfile` 实体字段是否满足暖通点位定义需求（键名、显示名、单位、数据类型、Modbus 参数、变换管道）。
- 实现 `ApplyToDevice` 核心逻辑：读取模板点位 → 生成 `CollectionPoint` → 记录已应用关系。
- 补充模板相关 CRUD API。
- 明确模板修改后是否影响已应用设备（建议第一阶段不做级联更新）。
- 支持从已有设备的点位反向生成模板。
- 增加模板应用和映射的最小测试用例。

### 6.4 非功能需求

- 模板应用操作应在一次数据库事务中完成，避免部分点位创建成功部分失败。
- 映射规则的变更应即时生效（直接作用于数据接入管道）。
- 模板数据结构应为后续控制命令模板和能耗标记预留扩展空间。
- 模板不得破坏已有设备的点位配置，应用逻辑必须明确覆盖或跳过策略。

## 7. 验收标准

### 7.1 模板 CRUD

- Given 已登录用户
  When 创建设备类型模板并添加 5 个遥测点位定义
  Then 系统应保存模板和点位，返回统一结构。

- Given 模板已关联到产品
  When 删除模板
  Then 系统应提示关联关系或阻止删除。

### 7.2 模板应用

- Given 已存在采集设备和完整的设备类型模板
  When 将模板应用到设备
  Then 系统应为模板中每个点位生成对应的 `CollectionPoint` 记录。

- Given 设备已有同名点位（相同 TargetName）
  When 再次应用模板
  Then 系统应按配置策略跳过或覆盖。

### 7.3 数据映射

- Given 已配置 JSON 字段 `temp` → 遥测键 `temperature` 的映射规则
  When 边缘网关上传 `{"temp": 24.5}`
  Then 系统应以遥测键 `temperature` 存储数据。

- Given 已配置映射规则
  When 上传的 JSON 中包含未映射字段
  Then 未映射字段应忽略或记录调试日志。

## 8. 不在范围内

- 不在本模块实现模板版本管理和历史追溯。
- 不在本模块实现模板市场和在线模板导入/导出。
- 不在本模块实现复杂映射表达式（嵌套 JSON 提取、正则替换、脚本转换）。
- 不在本模块实现模板修改后的自动级联更新（第一阶段不做）。
- 不在本模块实现完整的控制命令模板机制。

## 9. 依赖与后续关系

- 依赖"网关与设备接入模块"提供设备管理能力。
- 依赖"透传网关 Modbus 采集模块"的 `CollectionPoint` 实体。
- 依赖"边缘网关 JSON 接入模块"的数据接入管道用于映射。
- 本模块是"暖通业务设备建模模块"的点位定义基础。
- 本模块是"控制命令与审计模块"的控制点位定义基础。

**状态：实体已有，需补齐业务逻辑和 API。**

核心实体已存在，但模板应用逻辑、CRUD API 和前端管理界面需要从零或从半成品状态构建。建议优先完成 `ApplyToDevice` 核心逻辑和最基本的 CRUD API，数据映射规则先做最简单的 1:1 字段对字段映射。
