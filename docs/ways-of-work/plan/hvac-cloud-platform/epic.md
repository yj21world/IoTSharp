# Epic PRD：暖通空调云端物联网平台长期产品需求

## 1. Epic 名称

暖通空调云端物联网平台长期产品需求

## 2. 目标

### 问题

当前仓库是基于开源 IoTSharp fork 后形成的半成品物联网系统，已经经过一轮优化，减少了一部分多余和不必要的代码，但仍保留了一些后续需要继续清理的冗余代码和预留模块。当前系统的功能还不完善，尚不能直接支撑真实暖通空调项目的云端接入、数据采集、设备控制和运维分析。目标业务也不是泛物联网演示系统，而是面向暖通空调系统的云端管理平台，主要接入主机、水泵、阀门、电表、传感器、IO 模块等现场设备。现场设备通常通过工业网关接入，底层协议和数据结构偏工程通信视角，无法直接满足现场维护人员对设备状态、能耗、故障和控制的使用需求。

### 解决方案

在保留 IoTSharp 现有基础能力的前提下，逐步建设一个面向暖通空调场景的云端物联网平台。平台需要支持通过 HTTP、MQTT 等协议接入设备和网关，能够在云端查看实时数据与历史数据，并支持向现场设备下发控制命令。平台需要同时支持两类网关：一类是透传网关，网关只负责云端与设备之间的数据转发，设备采集、解析、存储和控制逻辑主要在云端配置和执行；另一类是具备边缘采集计算能力的网关，由网关主动采集下级 Modbus 设备、IO 设备，并将整理后的 JSON 数据上传到云端。

### 影响

长期目标是把当前 fork 仓库从一个功能不完整的通用物联网系统，演进为可用于暖通空调项目交付的云端平台。平台应能完成真实现场的网关接入、数据采集、实时监控、历史查询、控制下发、设备聚合建模和能耗统计，并为后续告警、自动化策略、租户角色权限、组态功能和高级能效分析打好基础。第一阶段应优先打通稳定的数据与控制闭环，而不是过早投入复杂组态和通用 SaaS 管理功能。

## 3. 使用对象说明

本平台主要面向暖通空调项目的云端接入、现场调试、设备运维、能耗统计和系统集成场景。长期需求文档不重点展开用户角色，具体角色、权限、菜单和操作边界放到后续平台管理模块中单独设计。

## 4. 高层业务流程

### 4.1 站点与网关接入

1. 管理员创建租户、客户或项目，并建立站点信息。
2. 管理员或调试工程师创建设备网关，并选择网关类型：透传网关或边缘采集网关。
3. 网关通过 MQTT、HTTP、访问令牌、设备密码或后续证书方式接入云端。
4. 平台记录网关在线状态、最后心跳时间、通信方式和关联的下级设备。
5. 运维人员可以查看网关是否在线，以及其负责的设备范围。

### 4.2 透传网关采集

1. 调试工程师在云端配置 Modbus 通信参数、从站地址、功能码、寄存器地址、数据类型、字节序、倍率、偏移量、单位和采集周期。
2. 云端按照配置生成 Modbus 请求数据。
3. 云端通过 MQTT 将请求发送给透传网关。
4. 透传网关将 Modbus 数据原样或按约定格式转发给现场设备，例如水泵变频器。
5. 现场设备返回 Modbus 响应，透传网关再把响应转发给云端。
6. 云端负责校验、解析、换算、归一化和存储数据。
7. 运维人员在平台中查看实时数据和历史趋势。

### 4.3 边缘采集网关数据上传

1. 具备边缘采集能力的网关在本地主动采集 Modbus 设备、IO 设备和传感器数据。
2. 网关将采集结果组装成 JSON 数据。
3. 网关通过 MQTT 或 HTTP 将 JSON 上传到云端。
4. 云端校验 JSON 格式，并根据配置将字段映射为平台遥测数据或设备属性。
5. 平台写入最新值和历史数据。
6. 运维人员通过统一设备视图查看数据，无需关心底层采集细节。

### 4.4 云端控制下发

1. 有权限的用户在平台上选择控制动作，例如启动水泵、停止水泵、设置频率、打开阀门、关闭阀门、复位故障等。
2. 平台检查用户权限、设备状态、控制模板和安全约束。
3. 对透传网关，平台生成底层控制数据，例如 Modbus 写寄存器命令，并通过 MQTT 下发给网关。
4. 对边缘采集网关，平台下发结构化 JSON 控制命令或网关约定格式的控制命令。
5. 平台跟踪命令创建、发送、确认、成功、失败、超时等状态。
6. 控制操作需要记录审计日志，便于追溯和排障。

### 4.5 暖通设备聚合建模

1. 现场底层接入对象可能是 Modbus 从机、IO 模块、电表、传感器或变频器。
2. 平台需要把一个或多个底层数据源聚合成运维人员关心的业务设备。
3. 例如水泵设备可能需要同时关注水泵变频器数据、电表数据、压力数据和流量数据。
4. 主机设备的信息可能来自多个 Modbus 从机、多个寄存器区域或外部计量设备。
5. 运维人员最终看到的是水泵、主机、阀门、传感器等业务对象，而不是零散的通信点位。

### 4.6 能耗统计

1. 平台接收来自电表、变频器、主机、水泵和传感器的能耗相关数据。
2. 平台按站点、设备类型、能源类型、设备分组和业务系统对数据进行分类。
3. 平台计算小时、日、月、年等周期统计数据。
4. 用户可以查看站点级、系统级、设备级能耗趋势。
5. 后续可扩展 COP、单位冷量电耗、异常能耗识别等高级指标。

## 5. 业务需求

### 功能需求

- 平台必须支持通过 HTTP、MQTT 等协议接入设备和网关。
- 平台必须保留并逐步完善 IoTSharp 已有的租户、客户、设备、产品、遥测、属性、规则、告警、API 和 UI 基础能力。
- 平台必须将网关作为一类核心设备进行管理，并支持网关与下级设备、逻辑设备之间的关联。
- 平台必须支持透传网关模式，云端负责复杂采集配置、Modbus 数据生成、响应解析、数据存储和控制命令生成。
- 平台必须支持边缘采集网关模式，网关上传 JSON 数据，云端负责解析、映射和存储。
- 平台必须支持 Modbus 类设备点位配置，包括从站地址、功能码、寄存器地址、寄存器数量、数据类型、字节序、倍率、偏移量、单位和采集周期。
- 平台必须支持遥测点位定义，使业务点位与底层协议细节解耦。
- 平台必须支持实时最新值查看。
- 平台必须支持历史数据按设备、业务设备、遥测键和时间范围查询。
- 平台必须支持控制命令定义，例如启动、停止、复位、设定值写入、阀门开关、频率设置等。
- 平台必须跟踪控制命令生命周期，包括已创建、已发送、已确认、成功、失败、超时和取消。
- 平台必须支持底层通信设备与业务暖通设备分离建模。
- 平台必须支持多个数据源聚合成一个业务设备，例如把水泵变频器和电表聚合为水泵设备视图。
- 平台必须支持暖通设备分类，包括主机、水泵、冷却塔、阀门、风柜、风机盘管、电表、温度传感器、压力传感器、流量传感器和 IO 模块等。
- 平台必须支持能耗数据分类和统计，首期优先支持电耗统计。
- 平台必须允许逐步清理当前仓库中保留的冗余代码和不必要模块，但不能阻塞核心业务功能交付。
- 平台新增 API 必须遵循统一返回结构 `ApiResult<T>`，列表和查询结果使用 `{ total, rows }`。
- 平台必须支持产品模板或设备模板机制，以便复用设备点位、解析规则和控制命令配置。
- 平台必须记录网关连接事件、数据接入失败、解析失败、控制下发、用户操作等关键审计信息。
- 平台必须为后续告警规则、自动化策略、报表、第三方系统对接和组态模块预留扩展空间。

### 非功能需求

- 平台必须采用渐进式二次开发策略，不允许完全重写现有系统。
- 平台必须明确区分实时接入、长周期采集、控制执行、数据解析和统计计算职责。
- 平台需要适应现场网络不稳定、网关掉线、设备响应超时等工业现场常见情况。
- 对透传网关场景，云端在解析失败时应保留必要的原始数据或错误上下文，便于调试。
- 控制下发必须具备权限校验、审计记录、超时处理和失败可见性。
- 多租户隔离是长期要求，即使租户、用户、菜单、角色系统放在后续完善，也不能破坏未来扩展路径。
- 对稳定核心概念应优先显式建模，不应完全依赖脚本或非结构化配置。
- 第一阶段优先保证正确性、可观察性和可维护性，再考虑高并发和大规模性能优化。
- API 和前端行为应保持一致，避免不同模块返回结构和交互方式分裂。
- 平台应部署在云端，不要求云端直接访问现场局域网设备，现场通信通过网关完成。
- 后续协议扩展应尽量不影响设备建模、遥测存储和业务视图。
- 产品设计应以中文暖通项目交付、调试和运维场景为主要语境。

## 6. 成功指标

- 网关可以在云端注册、认证、上线并显示在线状态。
- 透传网关可以完成完整闭环：云端生成 Modbus 请求、MQTT 下发、网关转发、设备响应、网关回传、云端解析、最新值更新、历史数据存储。
- 边缘采集网关可以完成完整闭环：JSON 上传、格式校验、字段映射、最新值更新、历史数据存储。
- 至少实现一个水泵场景，将变频器数据和电表数据聚合为业务水泵视图。
- 至少实现一个主机场景，将多个底层数据源聚合为业务主机视图。
- 授权用户可以下发至少一种安全控制命令，并查看执行结果。
- 用户可以按业务设备和遥测点查询实时数据和历史数据。
- 用户可以按站点、设备和时间范围查看基础电耗统计。
- 调试人员可以看到数据接入失败、解析失败和控制失败的原因。
- 后续模块 PRD、架构文档、实施计划和 AI 开发任务可以基于本长期 PRD 持续拆分。

## 7. 不在范围内

- 第一阶段不完整重做租户、用户、菜单、角色和权限系统，但需要保留后续完善空间。
- 第一阶段不实现完整组态功能，组态模块作为后续独立 Epic 规划。
- 第一阶段不实现复杂拖拽式大屏或自由仪表盘编辑器。
- 第一阶段不实现 AI 优化、预测性维护和自动节能策略。
- 第一阶段不追求支持所有工业协议，优先围绕 Modbus 和暖通常见接入方式落地。
- 不允许从零重写整个平台。
- 第一阶段不要求彻底清理所有继承自原项目的无用代码，清理工作应结合模块边界逐步进行。
- 移动 App、微信小程序、客户门户不属于当前 Epic 范围。
- 商业计费、订阅套餐、SaaS 收费系统不属于当前 Epic 范围。

## 8. 业务价值

**业务价值：高**

该 Epic 定义了当前 IoTSharp fork 的长期产品方向，使仓库从半成品通用物联网系统逐步演进为面向暖通空调项目的云端平台。它聚焦最高价值闭环：网关接入、数据采集、实时数据、历史数据、控制下发、设备聚合建模和能耗统计。同时，它也为后续模块级 PRD、技术架构、实施计划和 AI 辅助开发任务提供统一依据，减少后续开发中的反复讨论和方向漂移。

## 9. 产品阶段规划

### 阶段一：核心数据与控制闭环

- 网关注册、认证和在线状态。
- MQTT 和 HTTP 数据接入基础。
- 透传网关 Modbus 请求与响应闭环。
- 边缘采集网关 JSON 上传闭环。
- 最新遥测和历史遥测存储。
- 基础控制命令定义、下发、确认、超时和审计。
- 水泵、主机、电表、阀门、传感器等基础设备分类。

### 阶段二：暖通设备建模

- 逻辑业务设备与底层通信设备分离。
- 多数据源聚合为一个业务设备。
- 基于产品模板或设备模板复用点位配置。
- 水泵、主机、阀门、电表、传感器详情视图。
- 点位校验、解析调试和现场调试工具。

### 阶段三：能耗与运维

- 能耗遥测分类。
- 小时、日、月、年统计。
- 设备级、系统级、站点级能耗对比。
- 通信告警、阈值告警、异常值告警和控制失败告警。
- 运维报表和导出接口。

### 阶段四：平台完整性

- 租户、用户、角色、菜单和权限完善。
- 规则引擎和自动化策略优化。
- 系统可观测性、审计和维护工具完善。
- 冗余代码、预留模块和无用功能的持续清理。

### 阶段五：组态与高级能力

- 组态功能。
- 自定义看板。
- 高级能效指标。
- 预测性维护和优化建议。
- 更多协议适配和外部系统集成。

## 10. 假设与待确认问题

- 首期真实落地优先围绕 Modbus 设备和暖通空调场景。
- 透传网关与云端之间优先使用 MQTT 通信。
- 边缘采集网关上传 JSON 时，HTTP 和 MQTT 都应作为长期支持方向。
- 能耗统计首期优先支持电耗，水耗、冷量、运行时长和能效指标后续扩展。
- 透传网关场景优先保证云端配置能力，即使这会增加云端配置复杂度。
- 租户、角色、菜单和组态需求后续单独形成 Epic。
- 站点、项目、客户之间的层级关系需要在架构文档阶段进一步确认。
- 边缘网关 JSON 标准格式需要在模块级 PRD 中进一步定义，并考虑版本兼容。
- 控制命令的安全策略需要在进入生产环境前进一步细化。
