## 产品与物模型体系

  ### 一、产品（Produce）

  定位：设备量产模板，定义同类设备的默认配置。

  核心实体 (IoTSharp.Data\Produce.cs):

  • Name / Icon / Description — 基本信息
  • GatewayType / GatewayConfiguration — 网关配置
  • DefaultIdentityType / DefaultDeviceType / DefaultTimeout — 创建设备时的默认值
  • DefaultAttributes (List<ProduceData>) — 产品级默认属性，创建设备时自动复制到设备
  • Dictionaries (List<ProduceDictionary>) — 字段字典（技术名、显示名、单位、数据类型等）
  • ProduceToken — 用于 HTTP 遥测/属性上报的认证标识

  辅助实体：

  • ProduceData — 继承 DataStorage，存储键值对（支持 Boolean/String/Long/Double/DateTime/Json/XML/Binary）
  • ProduceDictionary — 字段元数据定义（KeyName, DisplayName, Unit, DataType, Place0-5 等）
  • ProduceDataMapping — 产品抽象字段到设备物理字段的映射，支持同一产品下不同设备使用不同字段名

  --------

  ### 二、物模型（DeviceModel）

  定位：命令模板定义，目前仅覆盖"服务/命令"要素。

  核心实体 (IoTSharp.Data\DeviceModel.cs):

  • ModelName / ModelDesc / ModelStatus — 基本信息
  • DeviceModelCommands — 关联命令列表

  命令实体 (IoTSharp.Data\DeviceModelCommand.cs):

  • CommandName / CommandTitle / CommandI18N
  • CommandType / CommandParams (JSON) / CommandTemplate

  --------

  ### 三、实现完整度

   层次         │ Produce                           │ DeviceModel                 │ 说明
  ──────────────┼───────────────────────────────────┼─────────────────────────────┼─────────────────────────────────────
  ─────────────────────────────────────────────────────────
   实体定义     │ 完整                              │ 简单                        │ Produce
  有属性/字典/映射；DeviceModel 只有命令
   数据库       │ 完整                              │ 完整                        │ 均有 DbSet 和多数据库迁移文件
   API 控制器   │ 丰富                              │ 基础                        │ ProducesController 有 CRUD +
  属性/字典/映射/遥测上报；DeviceModelController 只有 CRUD + 命令
   DTO          │ 完整                              │ 完整                        │ 均覆盖
   服务层       │ 缺失                              │ 缺失                        │ 业务逻辑直接写在控制器中，无独立
  Service
   物模型四要素 │ 属性✅ 遥测✅(映射) 服务❌ 事件❌ │ 属性❌ 遥测❌ 服务✅ 事件❌ │ 被拆分到两个体系，无统一物模型

  关键缺陷：没有标准物联网平台中统一的"物模型"实体（属性+遥测+服务+事件）。Produce 管属性/遥测，DeviceModel
  管命令，两者互不相干。

  --------

  ### 四、与设备（Device）的关系

  关系结构：

    Device
    ├── DeviceModelId (Guid?)  →  DeviceModel (直接外键，用于命令)
    └── ProduceId (Guid?)      →  Produce (间接外键，通过导航属性/迁移，用于模板)

    Produce (1) ──────► (N) Device          [Produce.Devices 导航属性]
    Produce (1) ──────► (N) ProduceData     [默认属性]
    Produce (1) ──────► (N) ProduceDictionary [字段字典]
    Produce (1) ──────► (N) ProduceDataMapping [数据映射]

    DeviceModel (1) ──► (N) DeviceModelCommand [命令定义]

  关键细节：

  • Device.cs 实体类中显式定义了 DeviceModelId，但没有显式定义 ProduceId（IoTSharp.Data\Device.cs:48）。
  • 然而数据库迁移中确实存在 ProduceId 列（20220914112335_Produce.cs:12），外键由 EF Core 约定通过 Produce.Devices
  导航属性自动生成。
  • 创建设备时通过 AfterCreateDevice (IoTSharp.Data\Extensions\DeviceExtension.cs:40) 关联产品：复制
  DefaultDeviceType、DefaultTimeout、DefaultAttributes。

  在设备接入流程中的实际使用：

  • Produce：创建设备时继承默认配置；HTTP 遥测/属性上报可通过 ProduceToken 路由到产品下聚合设备。
  • DeviceModel：仅在命令下发场景使用
  (DeviceModelController.GetCommandsByDevice)，遥测/属性接入流程完全不涉及物模型校验或解析。

  --------

  ### 五、总结

  IoTSharp 当前的产品与物模型体系是分裂的：

  • Produce 是一个实用的"设备量产模板"，功能较完整，但不符合标准物模型语义。
  • DeviceModel 是一个极简的"命令模板"，缺少属性、遥测、事件定义。
  • Device 同时直接关联两者，但两者之间没有任何关系。

  如果目标是构建标准 HVAC 设备管理能力，建议将 Produce 与 DeviceModel 整合为统一的物模型体系。
